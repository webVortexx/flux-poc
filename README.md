# flux-poc — Kubernetes GitOps PoC with Flux and Helm

A hands-on proof of concept demonstrating a full GitOps workflow on Kubernetes using FluxCD and Helm, running on a SUSE Linux VM with minikube.

---

## Stack

| Layer | Tool |
|---|---|
| OS | SUSE Linux (AWS EC2) |
| Container runtime | Docker |
| Kubernetes | minikube (single-node) |
| GitOps operator | FluxCD v2 |
| Package manager | Helm v3 |
| Source of truth | This GitHub repository |

---

## Architecture

```
GitHub (flux-poc)
      │
      │  Flux polls every 1 minute
      ▼
flux-system namespace
  ├── source-controller       → watches this Git repo
  ├── kustomize-controller    → applies YAML resources
  ├── helm-controller         → manages Helm releases
  └── notification-controller → sends alerts

      │
      │  helm-controller renders chart + applies to cluster
      ▼
dev namespace
  ├── nginxapp (Deployment, 2 replicas)
  └── nginxapp-service (Service, NodePort)
```

---

## Repository Structure

```
flux-poc/
└── clusters/
    └── my-cluster/
        ├── kustomization.yaml          # tells Flux: scan flux-system + dev
        ├── flux-system/                # auto-generated by Flux bootstrap — do not edit
        │   ├── gotk-components.yaml
        │   ├── gotk-sync.yaml
        │   └── kustomization.yaml
        └── dev/
            ├── namespace.yaml          # dev namespace definition
            ├── kustomization.yaml      # tells Flux: only apply helmrelease.yaml
            ├── helmrelease.yaml        # HelmRelease — your app config lives here
            └── nginxapp-chart/         # Helm chart
                ├── Chart.yaml
                ├── values.yaml
                └── templates/
                    ├── deployment.yaml
                    └── service.yaml
```

## How GitOps Works Here

The golden rule: **Git is the only source of truth. Nobody touches the cluster directly.**

```
You edit a file → git commit → git push
        ↓
Flux source-controller detects the new commit
        ↓
Flux helm-controller reads HelmRelease spec
        ↓
Helm renders the chart using values from helmrelease.yaml
        ↓
Kubernetes API server applies the rendered YAML
        ↓
kubelet tells Docker to start the containers
        ↓
Pods are running in dev namespace
```

If someone manually deletes a deployment directly on the cluster, Flux detects the drift within 60 seconds and restores it automatically from Git.

## Prerequisites

- SUSE Linux VM (or any Linux VM)
- Docker installed and running
- minikube installed
- kubectl installed
- Helm v3 installed
- FluxCD CLI installed
- GitHub account with a personal access token (repo scope)

### Install tools on SUSE

```bash
# Docker
sudo zypper install -y docker
sudo systemctl enable docker --now
sudo usermod -aG docker $USER

# kubectl
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
sudo install -o root -g root -m 0755 kubectl /usr/local/bin/kubectl

# minikube
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Helm
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

# Flux CLI
curl -s https://fluxcd.io/install.sh | sudo bash
```

## Setup

### 1. Start minikube

```bash
minikube start --driver=docker
kubectl get nodes
# Should show 1 node with status Ready
```

### 2. Bootstrap Flux

```bash
export GITHUB_TOKEN=<your-github-pat>
export GITHUB_USER=<your-github-username>

flux bootstrap github \
  --owner=$GITHUB_USER \
  --repository=flux-poc \
  --branch=main \
  --path=./clusters/my-cluster \
  --personal
```

This creates the GitHub repo, installs Flux into the cluster, and connects them.

### 3. Verify Flux is running

```bash
kubectl get pods -n flux-system
flux get all
```

All 4 Flux controllers should be Running and Ready.

### 4. Clone the repo and push your app

```bash
git clone https://github.com/$GITHUB_USER/flux-poc
cd flux-poc
# Add your app files under clusters/my-cluster/dev/
git add .
git commit -m "add app"
git push
```

Flux picks it up within 1 minute automatically.

## Making Changes

### Scale replicas

Edit `clusters/my-cluster/dev/helmrelease.yaml`:

```yaml
values:
  replicas: 3    # change this
```

```bash
git add .
git commit -m "scale to 3 replicas"
git push
```

Flux + Helm applies the change. No `kubectl` or `helm upgrade` needed.

### Change the image version

Edit `helmrelease.yaml`:

```yaml
values:
  image:
    tag: 1.25
```

```bash
git add .
git commit -m "upgrade nginx to 1.25"
git push
```

## Useful Commands

```bash
# Force immediate sync without waiting 1 minute
flux reconcile kustomization flux-system --with-source

# Check sync status
flux get kustomizations --watch

# Check Helm release status
flux get helmreleases -n dev

# Check Flux logs
flux logs

# Check Flux health
flux check

# See all pods
kubectl get pods -n dev
kubectl get pods -n flux-system

# See Helm releases
helm list -n dev

# See Helm release history
helm history nginxapp -n dev
```

## Drift Detection Test

This demonstrates Flux self-healing:

```bash
# Manually delete the deployment — bypassing Git
kubectl delete deployment nginxapp -n dev

# Watch Flux restore it automatically within 60 seconds
kubectl get pods -n dev -w
```

Flux detects the cluster no longer matches Git and runs `kubectl apply` again. The deployment comes back without any manual intervention.
